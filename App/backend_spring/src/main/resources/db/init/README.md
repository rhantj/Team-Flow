# 이 디렉터리는 "완전히 새 DB를 처음 만들 때"만 쓰입니다

`docker-compose.yml`이 이 폴더를 Postgres의 `docker-entrypoint-initdb.d`로 마운트해서,
**데이터가 하나도 없는 빈 볼륨으로 컨테이너를 처음 띄울 때만** 01부터 순서대로 자동
실행됩니다. 이미 데이터가 있는 볼륨을 재사용해 컨테이너만 재시작하면 이 폴더는 전혀
실행되지 않습니다.

**이 저장소는 Flyway를 쓰지 않습니다.** 이미 존재하는(운영 중인) DB에 새 컬럼/테이블을
추가하는 스키마 변경이라면, 배포 전에 사람이 직접 `psql`로 `ALTER TABLE`을 실행해야 합니다
(`App/DEPLOY_OCI.md` 8절 참고). `spring.jpa.hibernate.ddl-auto=validate`가 이 수동 절차가
빠졌을 때 애플리케이션이 틀린 스키마로 기동하는 걸 막아주는 마지막 안전판이다.

새 스키마 변경을 추가할 때는 여기(`db/init/`)에도 같은 내용의 파일을 다음 번호로 추가하세요
— 이 폴더가 "완전히 새로 만드는 DB(로컬 최초 기동, 신규 환경)"의 유일한 스키마 소스이기
때문에, 여기 반영하지 않으면 새 DB에서는 그 컬럼이 영원히 생기지 않습니다. 기존 운영 DB에
적용하는 수동 `ALTER TABLE`과 별개로, 이 파일도 함께 추가해야 두 경로가 어긋나지 않습니다.

09(`09_drop_legacy_field.sql`)는 짝이 되는 `docs/db/migrations/011_drop_legacy_field.sql`과
100% 같은 스키마 상태를 유지해야 합니다 — 완전히 빈 볼륨(신규 DB)에서만 이 파일이 자동
실행되는데, 여기서만 `field`가 rename되고 기존 운영 DB는 그대로면 신규 환경과 운영 환경의
스키마가 갈라지기 때문입니다. 그래서 두 파일 모두 `RENAME`/`DROP` 구문을 주석 처리한 채
보류 중입니다(`App/DEPLOY_OCI.md` 8-1절). 재활성화할 때는 반드시 두 파일을 함께 주석
해제할 것 — 한쪽만 풀면 다시 스키마가 갈라집니다.
