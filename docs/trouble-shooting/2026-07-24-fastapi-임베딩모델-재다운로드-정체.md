# FastAPI 기동이 임베딩 모델 재다운로드에서 멈추는 문제

- 날짜: 2026-07-24
- 브랜치: `feature/ai_assistent`
- 관련: `App/backend_fastapi/app/main.py`(lifespan `preload_embedding_model`), `App/docker-compose.yml`

## 증상

`docker compose up -d --build` 후 `backend-fastapi` 컨테이너가 `Up`이지만
`/openapi.json` 호출이 HTTP 000(응답 없음)이었다. 로그는 다음에서 멈춰 있었다.

```
INFO:     Started server process [1]
INFO:     Waiting for application startup.
```

`Application startup complete`가 4분이 지나도 나오지 않았다.

## 오해하기 쉬운 지점

startup이 멈춘 게 코드 결함처럼 보이지만 아니다. lifespan이 `preload_embedding_model()`에서
임베딩 모델(`rhantj/bge-m3-workflow-query-robust`, safetensors 약 2.3GB)을 내려받는 동안
블로킹한다. 정상 동작이며, 다운로드가 끝나면 startup이 완료된다.

## 원인

두 가지가 겹쳤다.

1. **캐시 볼륨이 없다.** compose의 `backend-fastapi`에 HuggingFace 캐시
   (`/root/.cache/huggingface`) 영속 볼륨이 없어, `--build`로 이미지를 새로 만들 때마다
   컨테이너의 모델 캐시가 초기화된다. 즉 재빌드마다 2.3GB를 다시 받는다.
2. **다운로드가 정체됐다.** 캐시 크기를 20초 간격으로 관찰하니 1.18MB에서 멈춰 증가하지 않았다.
   HF CDN 연결이 stale해진 것으로 보인다. 인증 실패는 아니었다(토큰 주입 정상, `.incomplete`
   blob 생성됨).

## 조치

정체된 연결을 리셋하기 위해 컨테이너만 재시작했다.

```bash
docker compose restart backend-fastapi
```

hf_hub 다운로드는 `.incomplete` blob에서 이어받으므로, 재시작 후 다운로드가 재개되어
2.29GB까지 완료됐고 startup이 끝났다.

```
INFO:     Application startup complete.
INFO:     Uvicorn running on http://0.0.0.0:8000
```

관찰 확인:

```bash
# 다운로드가 진행 중인지(정체인지) 캐시 크기 증가로 판별
docker exec workflow-backend-fastapi sh -c 'du -sb /root/.cache/huggingface | cut -f1'
```

## 근본 해결 (권장, 미적용)

`--build` 때마다 2GB 재다운로드 + 정체 위험을 없애려면 compose에 HF 캐시 영속 볼륨을 붙인다.

```yaml
services:
  backend-fastapi:
    volumes:
      - hf-cache:/root/.cache/huggingface
volumes:
  hf-cache:
```

이러면 최초 1회만 받고 이후 재빌드·재기동에서 캐시를 재사용한다. compose 파일 변경은
공유 인프라 수정이라 별도 승인 후 적용한다.

## 되돌리는 법

볼륨을 추가했다가 문제가 생기면 `volumes` 항목을 제거하고 `docker volume rm <name>` 후
재기동하면 원래(매번 재다운로드) 동작으로 돌아간다.
